# Contributing a Cloud CTF

Thanks for helping us expand [ctfs.html](ctfs.html) — the dedicated CTF page is one of the most useful learning resources on the site.

This guide covers how to add a new CTF challenge. For general contribution guidance see [CONTRIBUTING.md](CONTRIBUTING.md).

## TL;DR — the fast path

```bash
python3 tools/submit_ctf.py
```

The script prompts for everything (name, URL, description, section, tags), validates the URL for safety, generates the card HTML, inserts it into the correct section of `ctfs.html`, creates a git branch, and commits. Then open a PR.

## What counts as a CTF?

A resource belongs on **ctfs.html** if users can *play it* — interactively exploit something and capture flags, or deploy a deliberately-vulnerable environment and attack it. Examples of good fits:

- Browser-based challenges (e.g. [flAWS](http://flaws.cloud/), [Big IAM Challenge](https://bigiamchallenge.com/))
- Deployable vulnerable environments (e.g. [CloudGoat](https://github.com/RhinoSecurityLabs/cloudgoat), [EntraGoat](https://github.com/Semperis/EntraGoat))
- Scenario-based competitions (e.g. [Wiz Cloud Security Championship](https://www.cloudsecuritychampionship.com/))

**Not a good fit for the CTF page:**

- Passive training material (videos, slides) → add to [resources.html](resources.html) instead
- Generic security tools (scanners, linters) → `security-tools` section of resources.html
- Certification prep content → `certifications` section of resources.html
- Closed/paid platforms where you can't self-host (unless notable — e.g. HackTheBox BlackSky) → usually resources.html

When in doubt: if someone can point at the URL and say *"here's a thing you exploit"*, it belongs on ctfs.html.

## Sections on ctfs.html

Pick the best-fitting section:

| Section ID | When to use |
|---|---|
| `wiz-championship` | Monthly Wiz Cloud Security Championship challenges — reserved; use a calendar entry |
| `wiz-standalone` | Always-available Wiz CTFs outside the monthly Championship |
| `aws-ctfs` | Primarily AWS (IAM, S3, EC2, Lambda, etc.) |
| `azure-ctfs` | Primarily Azure / Entra ID |
| `gcp-ctfs` | Primarily GCP |
| `kubernetes-ctfs` | Primarily Kubernetes (any cloud) |
| `multi-cloud-ctfs` | Covers AWS + Azure + GCP in one deployment |
| `specialty-ctfs` | AI/ML, Secrets management, CI/CD pipeline security, or other niches |

If your CTF fits multiple clouds (e.g. Kubernetes Goat runs on any cloud), pick the *primary* focus and use tags to indicate the others.

## Card format

A CTF card on ctfs.html looks like this:

```html
<a href="https://example.com/your-ctf" class="card-link" target="_blank" rel="noopener noreferrer">
    <div class="resource-card"
        data-tooltip="Extended 2–3 sentence description that appears on hover. Cover what makes it unique, who benefits, and any prerequisites.">
        <img src="img/previews/example-com-your-ctf.jpg" alt="Preview" class="resource-preview" onerror="this.style.display='none'">
        <h3>Your CTF Name</h3>
        <p>Short 1-sentence description.</p>
        <div class="resource-tags">
            <span class="tag ctf">CTF</span>
            <span class="tag">AWS</span>
            <span class="tag">IAM</span>
        </div>
    </div>
</a>
```

### Required

- `href` — the canonical URL
- `<h3>` — the challenge name
- `<p>` — a 1-sentence description (~15 words)
- `<span class="tag ctf">CTF</span>` — always include this as the first tag

### Recommended

- `data-tooltip` — 2–3 sentences shown on hover. Specifies what makes the CTF unique, who it's for, and any prerequisites (AWS account, Docker, Terraform, etc.)
- `<img src="img/previews/...jpg" ... onerror="this.style.display='none'">` — preview image. The `onerror` handler lets the card render cleanly before the preview is auto-generated by CI
- Cloud + focus tags — e.g. `<span class="tag">AWS</span>`, `<span class="tag">IAM</span>`, `<span class="tag">Kubernetes</span>`

### Special tag classes

- `<span class="tag ctf">CTF</span>` — CTF marker
- `<span class="tag ai-security">AI/ML</span>` — AI/ML-focused CTFs
- `<span class="tag">...</span>` — all other tags

## Image preview

You don't have to generate a preview image yourself. When a PR with a new card lands, GitHub Actions runs `tools/generate_preview.py` which:

1. Reads the card's `href`
2. Checks `preview-mapping.json` to see if a preview already exists
3. If not, captures a screenshot via Playwright
4. Saves it as `img/previews/<slugified-domain-path>.jpg`
5. Adds the mapping to `preview-mapping.json`
6. Commits back to the PR branch

**Filename convention** — the slug is derived by lowercasing the domain + path and replacing non-alphanumerics with dashes. Example:

| URL | Image filename |
|---|---|
| `https://flaws.cloud/` | `flaws-cloud.jpg` |
| `https://github.com/Semperis/EntraGoat` | `github-com-semperis-entragoat.jpg` |
| `https://cloudhuntinggames.com/` | `cloudhuntinggames-com.jpg` |

Use the same filename in your card's `<img src="img/previews/....jpg">` so the preview lines up once generated. The `submit_ctf.py` script handles this automatically.

## Manual contribution (without the script)

If you'd rather edit by hand:

1. Open [ctfs.html](ctfs.html) in your editor
2. Find the right section (search for `id="aws-ctfs"`, `id="kubernetes-ctfs"`, etc.)
3. Add your card HTML just before the closing `</div></section>` of that section's `.resource-grid`
4. Save, commit, and open a PR

Example insertion point — look for the pattern:

```html
        <!-- AWS CTFs -->
        <section class="section" id="aws-ctfs">
            <h2>☁️ AWS CTFs</h2>
            <div class="resource-grid">
                <!-- ...existing cards... -->
                <a href="...">...</a>   ← add your new <a>...</a> right after this one
            </div>
        </section>
```

## PR checklist

Before submitting:

- [ ] URL returns HTTP 200 (or a documented expected redirect)
- [ ] Card includes the `CTF` tag as the first tag
- [ ] Card placed in the most appropriate section
- [ ] Description is 1 sentence, under ~20 words
- [ ] Tooltip (if included) is 2–3 sentences
- [ ] No secrets or personal info in the URL or description
- [ ] The CTF is genuinely playable (not just docs or marketing)

When you push a PR, CI will run:
- URL safety check (`tools/check_url_safety.py`)
- URL normalization (strip tracking params, resolve redirects)
- HTML validation
- Broken-link check (`lychee`)
- Auto-generated preview image (if missing)
